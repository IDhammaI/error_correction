# Commit 规则

本项目提交 commit 时遵循以下约定。

## 标题格式

使用 Conventional Commits 风格：

```text
type(scope): 中文概括
```

常用 `type`：

- `feat`：新增功能
- `fix`：修复问题
- `refactor`：重构代码，不改变业务行为
- `chore`：工程配置、文档、依赖、脚本等维护工作
- `docs`：仅文档变更
- `test`：测试相关变更

`scope` 优先使用受影响范围，例如：

- `frontend`
- `backend`
- `api`
- `router`
- `docs`

## 正文格式

如果提交内容超过一件事，正文使用中文无序列表：

```text
- 说明第一项变更
- 说明第二项变更
- 说明验证或影响范围
```

## 写法要求

- 标题要覆盖本次提交的主要范围，不要把范围写窄。
- 正文优先说明“做了什么”和“为什么做”，不要只罗列文件名。
- 一次提交尽量聚焦一个主题；小改动可以只写标题。
- 如果包含验证结果，可以在正文最后补一条，例如 `- 验证 npm run build 通过`。

## 示例

```text
refactor(frontend): 整理前端模块结构并补充中文注释

- 拆分 api 和 utils 单文件为按领域组织的模块目录，更新前端引用路径
- 为 components、composables、views 等前端核心代码补充中文注释，降低阅读和维护成本
- 优化 Toast、Modal 与工作台侧边栏偏移逻辑，避免认证页错误让出侧边栏宽度
- 补充本地开发启动文档和前端配置注释，记录前后端启动方式
```

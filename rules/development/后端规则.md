# 后端规则

## 技术栈与入口

- 后端是 `Flask` 纯 API 服务，入口文件是 `backend/web_app.py`。
- 路由通过 `Blueprint` 注册，统一由 `backend/routes/__init__.py` 挂载到 Flask app。
- 共享配置集中在 `backend/core/config.py`，使用 `pydantic-settings` 管理。
- 数据访问使用 `SQLAlchemy`，数据库入口在 `backend/db/__init__.py`。
- AI 工作流使用 `LangGraph`，主流程定义在 `backend/pipeline/workflow.py`。

## 目录分层

- `backend/routes/` 负责 HTTP 入参与响应，不承载大段业务编排逻辑。
- `backend/db/crud/` 负责数据库读写和查询封装。
- `backend/db/models.py` 负责 ORM 模型定义。
- `backend/core/` 负责全局配置、模型选择、额度、状态、邮件和运行存储等基础能力。
- `backend/pipeline/` 负责 OCR、分题、导出等工作流链路。
- `backend/agents/` 负责纠错、解题、讲解、笔记等 Agent 逻辑与 schema。
- `backend/tests/` 负责单元测试与集成测试，新增后端能力时优先在这里补测试。

## 模块副作用

- 禁止在模块顶层执行创建目录、写文件、启动连接等副作用操作。
- 目录创建优先通过 `core/config.py` 的显式初始化流程处理。
- `load_dotenv()` 只在入口或确实需要环境变量的模块中调用。
- 顶层允许的共享对象应是无状态或可控的单例，如配置对象、锁、工作流图，不要在导入阶段执行一次性业务逻辑。

## 路由与接口规则

- 新接口优先放入已有 `Blueprint`，按业务域归到 `auth`、`upload`、`questions`、`chat`、`notes`、`projects`、`settings`、`stats` 等模块。
- 路由层负责参数校验、登录态检查后的调用和响应格式整理，复杂业务逻辑下沉到 `core`、`pipeline`、`crud` 或 `agents`。
- `/api/*` 路由默认受登录校验保护，新增免登录接口时必须同步更新 `web_app.py` 中的豁免逻辑。
- 返回 JSON 时优先遵循现有响应结构，保持 `success`、`error`、`code`、业务数据字段风格一致。

## 数据库

- 使用 `with SessionLocal() as db:` 管理数据库会话生命周期。
- 写操作必须处理异常并在失败时 `rollback()`。
- ORM 到字典的转换优先使用现有序列化辅助函数。
- 数据库结构变更必须同步模型、迁移、CRUD、路由、前端调用和测试。
- 新增字段或表时，除了更新 `db/models.py`，还要同步 `db/__init__.py` 中的轻量迁移逻辑，避免老库启动失败。
- 对外暴露资源时优先复用现有 `public_id` 或 UUID 方案，不直接暴露内部自增主键。

## 配置与运行时状态

- 文件路径、运行目录、数据库路径、模型权重路径等统一通过 `core/config.py` 管理。
- 禁止硬编码运行时路径、模型路径、数据库路径或本机绝对路径。
- Provider 配置、系统托管模型、OCR 配置优先通过 `settings` 和现有加载逻辑解析，不要在路由里自行拼装环境变量。
- 请求级 provider 上下文要通过现有 context 和清理逻辑维护，避免线程复用导致串状态。

## 并发与会话状态

- 全局会话状态集中在 `backend/core/state.py`。
- 修改共享状态时必须使用对应锁保护。
- 锁内优先原地修改集合，避免重新赋值破坏共享引用。
- Agent 缓存初始化需要并发保护。
- 会话状态按 `user_id` 隔离，新增上传链路、OCR 缓存或运行时上下文时不能让不同用户共用同一份状态。
- 无状态对象才允许全局共享；带用户数据或请求上下文的数据必须做隔离。

## LLM 与 Agent

- Agent 调用入口优先使用现有封装函数。
- Provider、模型名和系统托管配置通过现有模型选择逻辑解析。
- 不同模型能力差异需要在适配层处理，不把分支散落到业务路由。
- 结构化输出、流式输出和 reasoning 输出变更必须同步测试。
- 模型选择优先复用 `core/model_selection.py`，不要在业务代码里直接跳过权限、来源或可用性校验。
- Agent schema、prompt、工具函数变更时，要同步检查调用方和测试是否仍兼容。

## OCR 与文件

- OCR 到 Agent 输入的转换以现有标准化函数为唯一入口。
- 新增 OCR block 类型时必须同步映射和测试。
- 文件名解析使用 `os.path.splitext()`。
- 上传文件使用安全随机文件名。
- 文件服务必须沿用现有安全路径拼接或等价校验，防止目录遍历。
- PaddleOCR 调用优先复用 `pipeline/paddleocr_client.py`，不要在其他模块重复实现任务提交、轮询和下载逻辑。
- 工作流里的 OCR、擦除、导出、错题保存等链路优先复用现有 `pipeline` 与 `workflow_run_store` 能力。

## 测试规则

- 后端测试放在 `backend/tests/`，使用 `pytest`。
- 新增或修改路由时，优先补对应路由集成测试或 CRUD 测试。
- 涉及结构化输出、模型选择、OCR 映射、工作流步骤或权限隔离的改动，必须同步补测试。
- 优先写稳定的单元测试、CRUD 测试和针对性集成测试，避免高成本、脆弱的大而全测试。

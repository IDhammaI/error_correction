# 提交规则

本项目提交 commit 时遵循以下约定。

## 标题格式

使用 Conventional Commits 风格：

```text
type(scope): 中文概括
```

常用 `type`：

- `feat`：新增功能
- `fix`：修复问题
- `refactor`：重构代码，不改变业务行为
- `chore`：工程配置、文档、依赖、脚本等维护工作
- `docs`：仅文档变更
- `test`：测试相关变更

`scope` 优先使用受影响范围，例如：

- `frontend`
- `backend`
- `api`
- `router`
- `docs`

## 正文格式

所有 commit 都必须包含正文。正文先用中文无序列表说明变更，再添加强制 `Checklist:` 段：

```text
- 说明第一项变更
- 说明第二项变更
- 说明验证或影响范围

Checklist:
- [x] 说明已完成的代码、文档或配置检查
- [x] 说明已完成的验证方式或无需额外验证的原因
```

## Checklist 强制要求

- 每个 commit message 必须包含 `Checklist:` 段。
- `Checklist:` 下至少包含一条 `- [x]` 已完成项。
- checklist 必须对应本次提交的真实完成状态，不能使用空泛占位内容。
- 涉及接口变更时，checklist 必须覆盖后端接口、前端调用和接口文档是否同步。
- 涉及跨端或跨模块变更时，checklist 必须说明相关代码、配置、文档或测试已经同步。
- 文档、配置、样式等小改动也必须包含 checklist；不能只写标题。

## 写法要求

- 标题要覆盖本次提交的主要范围，不要把范围写窄。
- 正文优先说明“做了什么”和“为什么做”，不要只罗列文件名。
- 一次提交尽量聚焦一个主题。
- 验证结果优先写入 `Checklist:`，例如 `- [x] npm run build 已通过`。

## 示例

```text
refactor(frontend): 整理前端模块结构并补充中文注释

- 拆分 api 和 utils 单文件为按领域组织的模块目录，更新前端引用路径
- 为 components、composables、views 等前端核心代码补充中文注释，降低阅读和维护成本
- 优化 Toast、Modal 与工作台侧边栏偏移逻辑，避免认证页错误让出侧边栏宽度
- 补充本地开发启动文档和前端配置注释，记录前后端启动方式

Checklist:
- [x] 前端引用路径已同步更新
- [x] 相关 UI 状态和布局影响范围已检查
- [x] 本地开发文档已同步
```

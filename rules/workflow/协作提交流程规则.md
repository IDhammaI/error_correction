# 协作提交流程规则

团队协作采用 fork-based workflow。

## 基本流程

- 开发者先 fork 主仓库到自己的账号或个人空间。
- 本地开发基于自己的 fork 仓库进行。
- 新功能、修复和重构都应创建独立分支，不直接在主分支开发。
- 完成本地验证后，将分支推送到自己的 fork 仓库。
- 从个人 fork 分支向主仓库发起 Pull Request。
- 主仓库通过 PR review 后再合并。

## 分支要求

- 分支名应能表达变更目的，例如 `feat/project-delete`、`fix/chat-provider`、`docs/rules-cleanup`。
- 一个分支尽量聚焦一个主题。
- 不把多个无关需求混在同一个 PR。
- 发 PR 前确认分支基于主仓库最新主分支，避免无意义冲突。

## PR 要求

- PR 标题使用中文概括本次变更。
- PR 描述需要说明变更内容、影响范围和验证结果。
- 涉及接口变更时，PR 描述必须说明后端接口、前端调用、测试和接口文档是否同步。
- 涉及数据库、配置、运行时路径或部署方式时，PR 描述必须说明迁移或兼容影响。
- PR 标题和描述不要带 AI IDE、AI 工具或具体工具品牌名，聚焦代码变更本身。
- 不提交 `.env`、数据库、构建产物、运行缓存和本地临时脚本。

## Review 要求

- 未经过 review 的代码不直接合入主仓库。
- review 重点关注行为正确性、接口契约、测试覆盖、回退风险和运行产物污染。
- review 反馈需要逐条处理；不接受的反馈应说明技术原因。
- 合并前应确认工作区干净，且提交符合 `workflow/提交规则.md`。
